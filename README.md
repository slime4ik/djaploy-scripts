# Что djaploy делает на твоём сервере

Прозрачность — главное, когда ты даёшь доступ к своему серверу. Здесь **дословно**
расписано всё, что [djaploy](https://djaploy.dev) выполняет на твоей машине: каждый
шаг, каждая команда. Никакой магии и чёрного ящика.

> Это не маркетинг, а распечатка реальных команд из нашего деплой-движка. Можешь
> сверить с тем, что видишь в живом логе деплоя — должно совпадать строка в строку.

---

## TL;DR

**Что мы делаем:** ставим fail2ban, Docker, клонируем твой репозиторий в `/opt/djaploy/`,
поднимаем контейнеры, настраиваем Caddy с автоматическим HTTPS. Опционально — отдельного
non-root пользователя, VPN, мониторинг.

**Чего мы НЕ делаем НИКОГДА:**
- ❌ не храним твой SSH-пароль (живёт только в оперативке во время деплоя);
- ❌ не храним твой `.env`, токены GitHub и пароли в нашей базе;
- ❌ не трогаем твои данные и тома (volumes) — БД и файлы остаются на месте;
- ❌ не удаляем ничего, кроме нашей же папки проекта при пересборке;
- ❌ не трогаем твой root-доступ и существующие SSH-ключи.

В любой момент можешь **отвязать** проект (мы перестаём им управлять, на сервере ничего
не удаляем) или **удалить полностью** (убираем только то, что разворачивали сами).

## Шаги деплоя — реальные команды

### 1. Подготовка сервера (fail2ban + базовый фаервол)
```bash
export DEBIAN_FRONTEND=noninteractive
apt-get update -y
apt-get install -y fail2ban curl ca-certificates git

# fail2ban: бан после 5 неудачных попыток SSH на 1 час
cat > /etc/fail2ban/jail.local <<'INI'
[sshd]
enabled = true
maxretry = 5
bantime = 1h
INI
systemctl enable --now fail2ban

# открываем только HTTP/HTTPS на локальном фаерволе (для Caddy и Let's Encrypt)
ufw allow 80/tcp ; ufw allow 443/tcp
```

### 2. Docker
```bash
# ставим официальный Docker, только если его ещё нет
command -v docker || curl -fsSL https://get.docker.com | sh

# зеркала реестра (чтобы образы тянулись стабильнее)
cat > /etc/docker/daemon.json <<'JSON'
{ "registry-mirrors": ["https://mirror.gcr.io", "https://dockerhub.timeweb.cloud"] }
JSON
systemctl restart docker
```

### 3. Клон твоего репозитория
```bash
mkdir -p /opt/djaploy
rm -rf /opt/djaploy/<твой-репо>     # чистим ТОЛЬКО нашу папку этого проекта
# токен передаётся в заголовке, НЕ в URL и НЕ в логах:
git -c http.extraheader="AUTHORIZATION: basic ***" clone --depth 1 \
    https://github.com/<твой-репо>.git /opt/djaploy/<твой-репо>
```
> Доступ к репозиторию — через GitHub App, который ты подключаешь сам и для выбранных
> репозиториев. Токен короткоживущий и в базу не пишется.

### 4. Переменные окружения
Твой `.env` пишется прямо на сервер (в нашу базу не попадает). Для веб-приложений
дополнительно подставляем безопасные значения:
```env
ALLOWED_HOSTS=<твой-домен>
CSRF_TRUSTED_ORIGINS=https://<твой-домен>
DEBUG=False
```

### 5. HTTPS через общий Caddy-шлюз (Let's Encrypt)
Один Caddy на весь сервер сам выпускает и продлевает SSL для всех твоих сайтов и
раскидывает запросы по нужным контейнерам через общую docker-сеть (алиас `app-<проект>`).
Поэтому на одном сервере спокойно живёт несколько проектов — за порты 80/443 они не
дерутся. Деплой без простоя: на время пересборки запросы придерживаются, а не падают в 502:
```caddy
# базовый /etc/caddy/Caddyfile просто подключает сниппет на каждый сайт:
import sites/*.caddy

# сниппет одного сайта (sites/<твой-домен>.caddy):
<твой-домен> {
    encode zstd gzip
    route {
        reverse_proxy app-<твой-проект>:<порт> {
            lb_try_duration 30s
            lb_try_interval 250ms
        }
    }
}
```
(полный пример со static/media, security-заголовками и защитой путей — в `scripts/Caddyfile.example`)

### 6. Сборка и запуск
```bash
cd /opt/djaploy/<твой-репо>
# поднимаем контейнеры проекта
docker compose up -d --build --remove-orphans
# подключаем контейнер к общей сети шлюза под алиасом app-<проект>,
# чтобы Caddy-шлюз нашёл его и проксировал
docker network connect --alias app-<твой-проект> djaploy <id-контейнера>
```

## Опциональные шаги (включаешь сам)

### Отдельный non-root пользователь
Чтобы повторные деплои и авто-деплой (CD) шли не под root:
```bash
useradd -m -s /bin/bash deploy
usermod -aG docker deploy           # доступ к Docker без sudo
mkdir -p /home/deploy/.ssh          # кладём отдельный ключ для деплоя
chown -R deploy:deploy /opt/djaploy/<твой-репо>   # передаём владение проектом
```
> Твой root-доступ при этом **не меняется** — мы его не трогаем.

### VPN (AmneziaWG) — закрыть админку/сервисы
```bash
apt-get install -y amneziawg amneziawg-tools
ufw allow 51820/udp
```
Приватные ключи VPN остаются **на твоём сервере** и в нашу базу не попадают.

### Откат версии
Откат переключает код на предыдущий коммит и пересобирает контейнеры.
**`.env` и тома (данные/БД) не трогаются** — остаются на месте.

---

## Как проверить, что это правда
1. Запусти деплой и смотри **живой лог** — там те же команды, что выше.
2. Зайди на свой сервер по SSH и проверь: `ls /opt/djaploy/`, `docker ps`,
   `systemctl status fail2ban`, `cat /etc/docker/daemon.json`.
3. Всё, что мы развернули, лежит в `/opt/djaploy/` и в стандартных местах Docker.

## Контроль остаётся у тебя
- Сервер, данные и код — всегда твои.
- Отвязал проект → мы перестали им управлять, ничего не удалили.
- Удалил проект → убрали только то, что сами разворачивали.
- Передумал — отозвал доступ GitHub App в один клик.
